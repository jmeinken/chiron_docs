Chiron Global Settings

All of these settings can be defined in your django settings file.

Website and API Settings

chiron.chiron_settings.CHIRON_SITE_TITLE = 'Chiron Test'

For the built-in website, the title is displayed in the Navbar and various other places.

chiron.chiron_settings.CHIRON_FOOTER_TEMPLATE = 'chiron/core/footer.html'

For the built-in website, allows a custom django template to use for the page footer.

chiron.chiron_settings.CHIRON_EXTRA_NAVBAR_ITEMS = None

For the built-in website, allows a custom django template to include addl links in nav menu.

chiron.chiron_settings.CHIRON_INFOBAR = 'This is a small dummy database for testing'

You can display an infobar at the top of all built-in views. For example, a warning bar that this is a test site. CHIRON_INFOBAR should have a string of the content to display and CHIRON_INFOBAR_TYPE should have the type of bar, which will usually be indicated by color:

"info" -> blue "warning" -> orange "danger" -> red

chiron.chiron_settings.CHIRON_INFOBAR_TYPE = 'info'

You can display an infobar at the top of all built-in views. For example, a warning bar that this is a test site. CHIRON_INFOBAR should have a string of the content to display and CHIRON_INFOBAR_TYPE should have the type of bar, which will usually be indicated by color:

"info" -> blue "warning" -> orange "danger" -> red

chiron.chiron_settings.CHIRON_LOGOUT_URL = '/accounts/logout'

The Chiron navbar has a logout link.

chiron.chiron_settings.CHIRON_SHOW_ANALYSIS_VIEW = True

Include the analysis view on the built-in UI

chiron.chiron_settings.CHIRON_AGG_DELIMITER = '; '

When aggregating result columns, use this delimiter for separating the values.

Data Dictionary Settings

chiron.chiron_settings.CHIRON_EVENT_CONCEPT_TYPE = 'date'

The event query system allows date concepts to be associated with specific collections (using collection.event_date_field and collection.event_end_date_field) indicating when an event occurred. Then users can do temporal queries at the collection level. In addition to date concepts, you can also substitute age in years (float) or age in days (int). However, since comparisons between collections are possible, all event concepts must use the same type of value. Set that type here.

OPTIONS: "date", "age_in_days", "age_in_years"

Mongo Database Settings

chiron.chiron_settings.CHIRON_MONGO_CONNECTION_SETTINGS = {'host': 'localhost'}

The MongoDB connection settings definition that will be passed to pymongo.

chiron.chiron_settings.CHIRON_USE_STAGING_DURING_ETL = False

Applies to python manage.py chiron_run_etl command. Normally, existing MongoDB collection(s) are deleted at the start of the ETL process and repopulated. If this setting is True, data will be loaded into separate staging tables and existing MongoDB collection(s) will be kept until the end of the ETL process. Upon successful completion of the ETL, the old collections will be replaced by the staging collections.

chiron.chiron_settings.CHIRON_MONGO_DATABASE_NAME_OVERRIDE = {}

Each dataset gets stored in its own MongoDB database. The database name is defined in the data dictionary Dataset.mongo_database_name field. If you want a particular Chiron deployment to use something different from what is defined in the data dictionary, you can use this setting. The dict key will be the dataset name and the dict value will be the desired MongoDB database name.

File Location Settings

chiron.chiron_settings.CHIRON_DATA_DICT_FIXTURE_PATH = 'chiron_config/fixtures.json'

For backing up and restoring the Chiron data dictionary (management commands chiron_backup_dd and chiron_restore_dd), where should the fixture file be stored.

chiron.chiron_settings.CHIRON_AUTOCREATE_SOURCE_LISTS = 'chiron_config.autocreate.autocreate_source_lists'

The path to where you dictionary of autocreate source lists is stored.

chiron.chiron_settings.CHIRON_PROCESSOR_MODULES = ['chiron.processors', 'chiron_config.processors']

Processor classes must be registered with Chiron. List any modules that load processors here to ensure that they are loaded during registry updates.

The recommended location for registering custom processors is in [your_project]/chiron_config/processors/__init__.py. Then this setting would need to be:

CHIRON_PROCESSOR_MODULES = ["chiron.processors", "chiron_config.processors"]

chiron.chiron_settings.CHIRON_DATA_SUMMARY_FUNCTION_PATH = 'chiron.views.views.get_data_summary_string'

You can customize what's shown on the built-in home page. This function will take a request object as an argument and should return an HTML string.

chiron.chiron_settings.CHIRON_SOURCE_DATA_DIRECTORY = ''

If loading source data from files, the directory path to prepend ex. C://Users/meinken/data/ ex. ../../data/

User and Permission Settings

chiron.chiron_settings.CHIRON_AGG_SUBJECT_COUNT_MIN_LIMIT = 5

For users with access_level="agg" (only allowed to see aggregated data), what is the minimum subject count they can see? Anything below that will show with a less-than sign. Zero subjects will still show as "0".

Performance and Cache Settings

chiron.chiron_settings.CHIRON_USE_CACHES = False

Chiron can cache various patient data and stats to improve performance. Setting to True will improve performance, but might cause problems in a development environment where you're actively modifying the data dictionary.

chiron.chiron_settings.MAX_CACHED_SUBJECT_IDS = 100000

Maximum number of Subject IDs that are allowed to be cached for a saved Cohort Definition. If a saved Cohort Definition has more Subject IDs than this number then it will not cache them.

chiron.chiron_settings.CHIRON_PRINT_QUERY_INFO = False

The query engine can print various statements as stdout that help with debugging query problems. This includes the amount of time it takes for each query step to run and notifications when a cache was used. Set to true to turn these statements on.

chiron.chiron_settings.CHIRON_ABBREVIATED_ETL_DEFAULT_RECORD_COUNT = ('read', 100)

A tuple with the count method and number of records to load for all source collections when using the abbreviated ETL.

("read", n)

Run until n records have been read, regardless of whether those records were loaded.

("write", n)

Run until n records have been loaded.

("match_full", n)

Only counts updates to an existing subject (i.e. adding data to a subject that was loaded in a previous step). Creation of new subjects can still happen (assuming if_no_match="create") but will not count toward the 100.

("match_strict", n)

Same as match full, but will not bother to create new subjects even when if_no_match="create".

chiron.chiron_settings.CHIRON_ABBREVIATED_ETL_RECORD_COUNTS = {}

The specific source collection record counts to import

CHIRON_ABBREVIATED_ETL_RECORD_COUNTS = {
    "my_source_collection": ("read", 500),
    "my_source_collection2": ("match_strict", 12),
}
